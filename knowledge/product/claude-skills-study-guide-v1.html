The Key Insight

Skills are the AI's "playbook" (methodology). MCP servers are the AI's "hands" (connectivity). Hooks provide deterministic automation. They're complementary, not competing. Most real workflows use all three together.

How They Work Together: A Real Example

Imagine you're deploying a web app. Here's how all the pieces collaborate:

1
You type: /deploy production

This triggers a Skill at .claude/skills/deploy/SKILL.md that contains your team's deployment checklist.

2
The skill instructs Claude to check CI status

Claude uses the GitHub MCP server to call the GitHub API and check if all CI checks are passing.

3
Claude runs the deploy command

A Hook fires on PreToolUse for the Bash tool, logging the command for audit purposes before it executes.

Part 3: Skills in Claude Code

Skills are the heart of Claude Code's extensibility. They transform Claude from a general-purpose AI into a specialist who knows your team's workflows, standards, and preferences.

What is a Skill?

A Skill is a reusable, filesystem-based resource that provides Claude with domain-specific expertise: workflows, context, and best practices. Unlike prompts (one-off instructions), skills load on-demand and persist across conversations.

Every skill lives in a directory and has one required file: SKILL.md. This Markdown file has two parts:

  1. YAML frontmatter (between --- markers) — tells Claude when to use the skill
  2. Markdown body — tells Claude what to do when the skill is active
code.claude.com 
---
name: explain-code
description: Explains code with visual diagrams and analogies. Use when
  explaining how code works, teaching about a codebase, or when the user
  asks "how does this work?"
---

When explaining code, always include:

1. **Start with an analogy** - Compare the code to something from everyday life
2. **Draw a diagram** - Use ASCII art to show flow, structure, or relationships
3. **Walk through step-by-step** - Explain what happens at each stage
4. **Highlight a gotcha** - What's a common mistake or misconception?

Keep explanations conversational. For complex concepts, use multiple analogies.
Critical Insight

The description field is the most important part of a SKILL.md file. Claude scans descriptions to decide relevance. If your description doesn't match what users naturally say, Claude will never auto-trigger your skill. Front-load the key use case — descriptions longer than 250 characters are truncated.

~/my-project $

---
name: review-pr
description: Review pull requests for bugs, security issues, and
code quality. Use when reviewing PRs, checking code changes,
or when the user mentions "review" or "PR".
---

# PR Review Checklist
When reviewing a PR, check these areas in order...
EOF

Choose where to put the skill

Personal skills (available across all projects): ~/.claude/skills/my-skill/
Project skills (shared with your team via git): .claude/skills/my-skill/

2

3

At minimum, include name and description. The description should include trigger keywords that match what users naturally say.

4

5

Auto-trigger: Ask Claude something that matches the description (e.g., "review this PR")
Manual trigger: Type /review-pr in Claude Code

Arguments & Dynamic Context Injection

Passing Arguments to Skills

Skills accept arguments via the $ARGUMENTS placeholder. When you run /fix-issue 123, Claude receives the skill body with $ARGUMENTS replaced by 123.

---
name: fix-issue
description: Fix a GitHub issue
disable-model-invocation: true
argument-hint: [issue-number]
---

Fix GitHub issue $ARGUMENTS following our coding standards.

1. Read the issue description
2. Understand the requirements
3. Implement the fix
4. Write tests
5. Create a commit

Access individual arguments with $ARGUMENTS[0], $ARGUMENTS[1], etc. — or the shorthand $0, $1, $2.

Dynamic Context with Shell Commands

Use !\`command\` syntax to run shell commands before the skill content is sent to Claude. This is preprocessing — Claude only sees the final output.

---
name: pr-summary
description: Summarize changes in a pull request
context: fork
agent: Explore
allowed-tools: Bash(gh *)
---

## Pull request context
- PR diff: !`gh pr diff`
- PR comments: !`gh pr view --comments`
- Changed files: !`gh pr diff --name-only`

## Your task
Summarize this pull request...

Built-in skills ship with Claude Code and are always available. They're prompt-based playbooks that can spawn parallel agents, read files, and adapt to your codebase. Custom skills extend this with your domain knowledge.

When to Build Custom

Build a custom skill when: the task involves knowledge specific to your context, your team does the same multi-step task more than twice a week, or you find yourself giving Claude the same instructions repeatedly.

The Hooks System

Hooks are shell commands that execute at specific lifecycle points in Claude Code. Unlike skills (which Claude can follow), hooks are deterministic — they always run when their trigger fires.

Hook Events

EventWhen It FiresExample Use
PreToolUseBefore Claude uses a toolValidate file paths before writes
PostToolUseAfter Claude uses a toolRun tests after file changes
NotificationWhen Claude produces a notificationSend desktop alerts
StopWhen Claude finishes a responseLog completion metrics

Configuration

Hooks are configured in .claude/settings.json:

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "command": "npm test 2>&1 | head -20"
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": "echo 'About to run bash command'"
      }
    ]
  }
}
Hooks vs Skills

Hooks are for enforcing rules deterministically (tests must run after writes). Skills are for providing guidance that Claude follows flexibly. If something must always happen, use a hook. If something should usually happen, use a skill.

CLAUDE.md Files: The Configuration Hierarchy

CLAUDE.md files are persistent instructions that Claude reads at the start of every session. They work like a constitution — always active, always loaded.

LevelLocationScopeUse For
EnterpriseManaged settingsAll users in orgSecurity policies, compliance rules
User~/.claude/CLAUDE.mdAll your projectsPersonal preferences, global style rules
Project./CLAUDE.mdThis repoProject-specific context, tech stack, conventions
Directory./src/CLAUDE.mdThis directory treeModule-specific rules for monorepos

Claude loads these in order: Enterprise → User → Project → Directory. More specific files override more general ones.

Part 4: Plugins in Claude Cowork

Cowork is where skills become operational. In Chat, skills influence replies. In Cowork, skills govern every file Claude creates, every process it follows, every output it delivers.

What is Cowork Mode?

Cowork is a research preview feature built into the Claude desktop app that brings agentic capabilities to knowledge work. Instead of responding to prompts one at a time, Cowork takes on complex, multi-step tasks and executes them autonomously.

Cowork vs Chat vs Code

Chat = conversational, one response at a time.
Code = command-line agent for developers, direct filesystem access.
Cowork = autonomous task execution for everyone, sandboxed VM, plugins.

Skills in Cowork: How They Work

Skills in Cowork are not the same thing as plugins. This distinction matters. A skill handles one task — it's a set of instructions that teaches Claude how to do something specific. A plugin is a bundle of multiple skills, connectors, commands, and sub-agents packaged together.

Skills vs Plugins in Cowork

Skill = a single SKILL.md file with instructions for one workflow (e.g., "write a blog post in our brand voice").
Plugin = a complete toolkit bundling multiple skills + MCP connectors + sub-agents (e.g., the "Marketing" plugin with content creation, SEO, campaigns, and brand review skills all together).

Cowork comes with built-in skills that activate automatically: Excel processing, Word documents, PowerPoint presentations, PDF processing. Claude uses these without you needing to install anything. But the real power comes from installing additional skills or creating your own.

Two Ways Skills Trigger in Cowork

MethodHow It WorksExample
Auto-triggerClaude recognizes when a skill applies based on your request and activates it automaticallyAsk "create a PDF report" and Claude loads the PDF skill
Explicit invocationType / in the chat to see available skills and select one manuallyType /deep-research to invoke that specific skill

Skills also adapt to the surface they're running on. A research skill may produce a Word document in Cowork, but detailed data breakdowns in Excel.

1. **Summary** - 2-3 sentence overview of the week
2. **KPIs** - Key metrics with week-over-week changes
3. **Accomplishments** - What got done (with specifics)

Common Upload Errors

Folder name doesn't match skill name: The ZIP's root folder name must match the name field in SKILL.md.
Files at ZIP root instead of in folder: Your SKILL.md must be inside a folder, not at the ZIP's top level.
Missing SKILL.md: Every skill folder must contain a SKILL.md file — it's the only required file.
Code execution disabled: Skills require code execution to be enabled in your org settings.

Part 5: Model Context Protocol (MCP)

MCP is the universal adapter that connects Claude to the outside world. Understanding it is essential because skills, plugins, and tool integrations all build on top of MCP.

What is MCP?

The Model Context Protocol is an open standard that enables developers to build secure, two-way connections between data sources and AI-powered tools. Think of it like USB-C for AI — one standardized protocol that connects to everything.

Building Custom MCP Servers

TypeScript Example 
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";

const server = new McpServer({
  name: "my-server",
  version: "1.0.0",
});

// Define a tool
server.tool("get-weather", { city: { type: "string" } }, async ({ city }) => {
  const data = await fetchWeather(city);
  return { content: [{ type: "text", text: JSON.stringify(data) }] };
});

// Start the server
const transport = new StdioServerTransport();
await server.connect(transport);
Python Example 
from mcp.server import Server
from mcp.server.stdio import stdio_server

server = Server("my-server")

@server.tool("get-weather")
async def get_weather(city: str) -> str:
    data = await fetch_weather(city)
    return json.dumps(data)

async def main():
    async with stdio_server() as streams:
        await server.run(streams[0], streams[1])

asyncio.run(main())

Part 7: API & Developer Ecosystem

Tool Use in the API

The Claude API supports tool use (function calling), enabling Claude to interact with external systems programmatically. You define tools with names, descriptions, and input schemas — Claude decides when to call them.

# Python example with Anthropic SDK
import anthropic

client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    tools=[{
        "name": "get_weather",
        "description": "Get the current weather in a location",
        "input_schema": {
            "type": "object",
            "properties": {
                "location": {"type": "string", "description": "City name"}
            },
            "required": ["location"]
        }
    }],
    messages=[{"role": "user", "content": "What's the weather in London?"}]
)

SKILL.md Complete Reference

---
# REQUIRED/RECOMMENDED
name: skill-name                    # Lowercase, hyphens, max 64 chars
description: |                      # Max 1024 chars. THE most important field.
  What this skill does and WHEN to use it. Include trigger keywords.
  Front-load the key use case (truncated at 250 chars in listings).

# INVOCATION CONTROL
disable-model-invocation: false     # true = manual only (/name)
user-invocable: true                # false = hidden from / menu
argument-hint: "[issue-number]"     # Shown during autocomplete

# EXECUTION
allowed-tools: Bash(npm *), Read    # Tools without permission prompts
model: claude-sonnet-4-6              # Model override
effort: high                        # low | medium | high | max (Opus only)
context: fork                       # Run in forked subagent context
agent: Explore                      # Subagent type (Explore, Plan, etc.)
shell: bash                         # bash (default) or powershell

# SCOPING
paths: ["src/**/*.ts", "*.config"]  # Glob patterns for activation
hooks:                              # Skill-scoped hooks
  PostToolUse:
    - matcher: Write
      command: "npm test"
---

# Skill Instructions

Your markdown instructions here. Claude follows these when the skill is active.

- Current branch: !`git branch --show-current`
- Changed files: !`git diff --name-only`

1. First step
2. Second step using $ARGUMENTS
3. Third step with $0 (first arg) and $1 (second arg)

Glossary

TermDefinition
SkillA markdown-based instruction package that teaches Claude domain-specific workflows. Lives in SKILL.md.
PluginA bundle of skills, commands, MCP servers, hooks, and sub-agents packaged for distribution.
MCPModel Context Protocol — an open standard for connecting AI to external tools and data.
MCP ServerA standalone program that exposes tools, resources, and prompts to Claude via MCP.
HookA shell command that runs at specific lifecycle events (PreToolUse, PostToolUse, etc.).
Slash CommandA manually-triggered prompt template. Legacy concept now merged into the Skills system.
CLAUDE.mdA persistent instruction file that Claude reads at session start. Works at user, project, or directory level.
FrontmatterYAML metadata at the top of a SKILL.md file (between --- markers) that controls behavior.
Sub-AgentA specialized agent instance that a plugin can delegate tasks to for focused work.
CoworkClaude Desktop's autonomous task execution mode for non-technical knowledge work.
ArtifactSelf-contained interactive content (React, HTML, visualizations) created by Claude in a side panel.
ConnectorA pre-built MCP integration available through the Claude Desktop UI.
Desktop ExtensionA .mcpb file that installs an MCP server with zero configuration.
Agent Skills StandardAn open specification at agentskills.io making skills portable across AI platforms.
